# Season Simulation Test Runner

Automated testing of the complete league season lifecycle: draft → regular season → playoffs → season completion.

## Quick Start

```bash
export SUPABASE_SERVICE_ROLE_KEY="your-service-role-key"
npm run test:simulation
```

Or directly:

```bash
node scripts/simulation-test-runner.mjs
```

## What It Tests

The test runner creates temporary leagues in the database, invokes the `process-week-results` edge function to simulate the full season, validates the results, and cleans up. Each test runs the complete lifecycle:

1. **Seed** — league, members, season, standings, draft picks, round-robin matchups, week snapshots (all with deterministic mock data)
2. **Regular season** — single edge function call processes all weeks, updates standings, transitions to playoffs
3. **Playoffs** — for each round (quarters → semis → finals): inject mock snapshots, call edge function, verify advancement
4. **Validate** — season completed, champion/runner-up recorded, standings math correct, all matchups scored
5. **Cleanup** — all test data deleted

### Negative / edge case tests
| Test | What it verifies |
|------|-----------------|
| IDEMPOTENT | Calling the edge function twice on the same league is a no-op the second time (team1_gain IS NULL filter) |
| FUTURE-DATES | Matchups with week_end in the future are never processed (week_end < now filter) |
| EMPTY-PORTFOLIO | A team with no holdings (no snapshots, no drafts) auto-loses every matchup |
| TIED-GAINS | Equal dollar + percent gains produce ties in regular season; playoff seed tiebreaker picks higher seed |

## Validation Checks

Each test validates:

- `season_status = 'completed'`
- `champion_user_id` and `runner_up_user_id` are set in `league_seasons`
- `completed_at` is not null
- `final_standings` JSON is populated
- All regular season matchups have `team1_gain`, `team2_gain`, and `winner_user_id`
- Bye week matchups: `team2_gain` is null, winner is team1
- Standings math: `wins + losses + ties = numWeeks` for every team
- Even teams: `SUM(wins) = SUM(losses)` across all teams
- Expected standings match (wins/losses per user computed from deterministic mock data)
- All non-placeholder playoff matchups have results
- Finals winner matches `champion_user_id`
- Finals loser matches `runner_up_user_id`

## How It Works

### Mock Data Design

**Regular season scoring:** Each team gets a deterministic gain based on their index. Team 0 (strongest) gets `numTeams * $10`, team N-1 (weakest) gets `$10`. Since the edge function loads all weeks' snapshots with identical mock prices, every user has the same gain every week. The higher-gain team always wins.

**Playoff scoring:** After each playoff round is generated by the edge function, the test runner injects `week_snapshots` for that round's matchups. The higher-ranked team (lower user index) gets `$50` gain, the weaker team gets `$20`.

**Week snapshots for all weeks:** Snapshots are created for every regular season week (not just week 1). This is required because the edge function loads snapshots for `leagueMatchups[0].week_number`, and without an `ORDER BY`, the first matchup returned could be from any week. Having snapshots for every week ensures deterministic scoring.

### Edge Function Call Pattern

Since all matchup dates are backdated (well in the past), the edge function processes everything it can find per call:

| Call | What gets processed |
|------|-------------------|
| 1 | ALL regular season matchups (weeks 1-N), standings updated, playoffs generated |
| 2 | First playoff round (quarters or semis), winners advanced to next round |
| 3 | Next playoff round (semis or finals) |
| 4 | Finals (8-team playoffs only, since quarters needed an extra round) |

Between calls, the test runner:
1. Queries the newly generated/populated playoff matchups
2. Ensures their `week_end` dates are in the past
3. Injects mock `week_snapshots` for that round
4. Calls the edge function

### Test Isolation

- Each test uses deterministic UUIDs based on test index (`11111111-0000-4000-a000-{index}`)
- Test user IDs follow `sim-t{testIndex}-user-{userIndex}` (plain text, no FK to `auth.users`)
- League names follow `__SIM_TEST_{configName}__`
- Cleanup runs before seeding (handles leftover data from failed runs) and after validation
- Tests run sequentially to avoid edge function cross-contamination

## Output

Console output shows per-test standings, playoff results, and pass/fail. A log file is written to `logs/simulation-test-{timestamp}.log` for review.

Example summary:
```
================================================================
  SUMMARY

```

## Prerequisites

- Node.js 18+ (uses native `fetch`)
- `@supabase/supabase-js` (installed as devDependency)
- `SUPABASE_SERVICE_ROLE_KEY` environment variable
- The `process-week-results` edge function deployed

## Important Notes

- Each edge function call passes the test league's `league_id` in the request body, so only the test league is processed. Real leagues are never affected.
- Tests take ~7 minutes total for all 18 configurations (network latency to Supabase + sleep delays between edge function calls).
- The test runner exercises the actual production edge function code path: snapshot-based scoring, standings updates, playoff bracket generation, winner advancement, and season completion via `complete_league_season` RPC.
